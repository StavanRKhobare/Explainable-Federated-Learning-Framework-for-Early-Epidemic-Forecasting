Project Overview

Why Epidemic Intelligence Matters

Dengue and malaria collectively infect hundreds of millions of people annually. Traditional surveillance systems detect outbreaks weeks after they begin — too late for effective intervention.

EpiGraph AI addresses this with a novel Split-Federated Spatio-Temporal GNN that learns across hospital networks without sharing any raw patient data, enabling real-time, district-level outbreak risk forecasting.

By combining GRU temporal modeling, Graph Attention Networks, and Explainable AI, the system tells health authorities not just where an outbreak will happen — but why.

Problem Statement

What Traditional Systems Get Wrong

Existing surveillance frameworks fail to provide early, precise, and privacy-preserving outbreak detection.

⏱️

Delayed Detection

Centralized reporting pipelines introduce 2–4 week delays. Outbreaks are confirmed only after widespread transmission has already occurred.

🏛️

Centralized Surveillance Limits

Aggregating raw patient data from all hospitals into a central server violates institutional privacy policies and healthcare regulations like DISHA.

🗺️

No Spatial Intelligence

Traditional models treat each district independently. They cannot model how disease spread across neighboring geographic borders, missing cross-district contagion signals.

⏩

Sequential Dilution

Standard RNNs struggle to model sharp 3-week biological incubation spikes. Long-term memory gets diluted across sequential timesteps, missing the mosquito breeding → infection lag.

🧪

Black-Box Predictions

ML models that predict outbreaks without explanation are rejected by health authorities. Clinicians need to know which factors — rainfall, temperature, case history — drove the alert.

📡

Offline Rural Clients

Rural district hospitals have intermittent connectivity. Synchronous centralized training fails catastrophically when a single "straggler" hospital loses connection mid-round.

Objectives

What EpiGraph AI Achieves

01

Early Outbreak Prediction

Predict district-level dengue/malaria outbreak risk 1–2 weeks before case explosion using the 3-week incubation window modeled by the Temporal Graph Attention layer.

02

Graph-Based Disease Spread Modeling

Use a Spatial DGAT/GAT to model how disease propagates across neighboring districts via geographic adjacency edges in the district graph.

03

Decentralized Privacy-Preserving Learning

Implement Split-Federated Learning so each hospital trains locally and only shares a 64-dim encrypted embedding — zero raw patient records cross network boundaries.

04

Explainability via Attention Weights

Surface temporal attention weights that explain which past weeks and which environmental features (rainfall, temperature) most influenced each outbreak alert.

05

District-Level Hotspot Identification

Generate a risk-ranked list of districts with color-coded alerts (High/Medium/Low) on an interactive map dashboard for district health officers and the Ministry of Health.

06

Asynchronous Federated Aggregation

Handle straggler hospitals via timeout-based asynchronous aggregation using Flower (flwr), ensuring the system is robust to rural connectivity limitations.

System Workflow

End-to-End Pipeline

Eight sequential stages from raw data collection to explainable outbreak alerts.

01
📥
Data Collection
Disease case data, NASA POWER weather API, GADM boundaries, and WorldPop population fetched per district.
02
🔧
Preprocessing
Normalize case counts per 100K population. Align weekly timestamps. Impute missing weather data using district-level interpolation.
03
⏱️
Temporal Encoding
Each client runs GRU over last 4 weeks for macro-trend + Temporal Graph Attention for 3-week incubation spike capture.
04
🔢
Embedding Generation
Feature Fusion MLP combines GRU and TGA outputs into a 64-dim privacy-safe local temporal embedding.
05
🕸️
Spatial Graph Learning
Central server receives all district embeddings. DGAT propagates signals across geographic adjacency edges to model cross-district spread.
06
🔗
Federated Aggregation
FedAvg aggregates spatial representations across all districts. Straggler timeout handles rural connectivity issues via async rounds.
07
📊
Prediction
District-level outbreak risk scores (0–1) generated. High-risk districts flagged. Gradients sent back to update local client models.
08
💡
Explainability Dashboard
SHAP values + attention weights surface which features (rainfall 3wks ago, temp spike) drove each alert. Displayed on Next.js dashboard.
Model Deep Dive

How Each Component Works

Click to expand detailed technical explanations of each AI layer.

🔄 GRU — Gated Recurrent Unit Temporal
▼

The GRU processes the 4-week sliding window of disease case counts + weather features as a sequential input. It maintains a hidden state h_t that acts as compressed memory of the seasonal trend.

Why GRU over LSTM?

GRU uses only two gates (reset, update) vs LSTM's three (input, forget, output), making it 30% more parameter-efficient for district-level data with limited samples.

What it captures

The 6-month seasonal macro-curve — slow rises and falls in dengue cases tied to monsoon patterns. The GRU "remembers" whether the district is in a rising or falling phase.

🕸️ Temporal Graph Attention — TGA Incubation
▼

Each week is treated as a node in a temporal graph. Attention edges are learned between time-steps, allowing the network to directly connect Week T to Week T-3 with a heavy attention weight — modeling the dengue 3-week incubation lag.

The Incubation Spike Problem

If a mosquito breeding event occurs in Week 1, cases explode in Week 4. A GRU alone dilutes this signal across 4 sequential steps. TGA creates a direct skip-connection: α(t, t-3) → high.

Relationship to Transformers

Technically a restricted Transformer where only epidemiologically-motivated edges (1-week, 3-week, 4-week lags) are attended to — not all pairs — reducing computational complexity significantly.

🌐 Spatial DGAT — Dynamic Graph Attention Spatial
▼

DGAT extends standard GAT by learning dynamic attention coefficients α_ij between nodes. Each district node aggregates signals from neighboring districts, weighted by learned relevance — allowing disease spread across borders to be modeled explicitly.

Why not standard GCN?

GCN uses fixed spectral weights. DGAT learns task-specific spatial attention — a district with high population density sharing a border with an outbreak district gets a higher attention coefficient than a sparsely-connected neighbor.

Server-side placement

Only the server can run DGAT because it needs embeddings from ALL districts simultaneously. Running DGAT locally would mean each hospital only sees its own embedding and its neighbors' — not the full national graph.

🌸 Split-Federated Learning Privacy
▼

This architecture combines Federated Learning (multiple clients, shared global model) with Split Learning (the model is split between client and server, with the client doing only the first few layers).

Training Loop

1. Client forward pass: Raw data → GRU → TGA → MLP → Embedding. 2. Embedding sent to server. 3. Server: DGAT → Loss. 4. Server computes ∂Loss/∂Embedding. 5. Gradient sent back. 6. Client backward pass updates local weights.

Privacy Guarantee

The MLP applies a non-linear ReLU projection before transmission. This makes it computationally infeasible to reverse-engineer the exact patient count from the 64-dim embedding without access to the local model weights.

🧩 Defence Q&A — Key Talking Points
Q: Can embeddings be reverse-engineered?
Non-linear ReLU MLP projection makes inversion computationally infeasible without local weights. Privacy preserved.
Q: Why not just a centralized DGAT?
Centralized training violates DISHA data regulations. Hospitals legally cannot share real-time admission data.
